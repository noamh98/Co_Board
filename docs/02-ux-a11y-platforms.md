### 17.3 Design-system assessment & proposal

#### 17.3.1 Current token state (verified, not assumed)

`docs/00-discovery.md` describes "tokens exist ad hoc in `tokens.css`" — code inspection shows the reality is more specific and more concerning:

- **`app/src/styles/tokens.css`** (101 lines) — defines spacing (4px grid, `--sp-1`…`--sp-12`), fluid typography via `clamp()` (`--tx-xs`…`--tx-2xl`, `--tx-cell`), radius (`--r-xs`…`--r-full`), shadows (`--sh-xs`…`--sh-xl`), touch-target sizes (`--cell-min: 92px`, `--target-min: 48px`, `--target-sm: 44px`), z-index scale, safe-area insets, transition presets — **and** a full green/blue "Claude Design 2026" color palette (`--cl-bg`, `--cl-primary: #1F7A5C`, category tints for Fitzgerald parts-of-speech).
- **`app/src/styles/themes.css`** (111 lines) — dark-mode (`@media (prefers-color-scheme: dark)` + `.dark-mode` class) and high-contrast (`.high-contrast` class) overrides **for the green palette above**, plus a `forced-colors: active` block.
- **`app/src/presentation/ui/tokens.css`** (228 lines) — a **second, later-loaded** file (comment at line 6: "חייב להיטען *אחרי* index.css כדי שינצח" = "must load after index.css to win") that redefines the exact same `--cl-*` color variable names with a **coral palette** (`--cl-primary: #E8694C`), and **independently reimplements its own dark-mode and high-contrast blocks** for coral, using the same selectors (`@media (prefers-color-scheme: dark) { :root:not(.light-mode):not(.high-contrast) }`, `.dark-mode:not(.high-contrast)`, `.high-contrast`) as `styles/themes.css`.
- Import order confirms this is not hypothetical: `app/src/index.css:9-10` imports `styles/tokens.css` then `styles/themes.css`; `app/src/main.tsx:8` imports `presentation/ui/tokens.css` **after** `index.css`. At equal CSS specificity, later-in-cascade wins — so **`styles/themes.css`'s dark-mode and high-contrast color values are dead code today**, silently overridden by `presentation/ui/tokens.css`. Non-color tokens (spacing/radius/typography/shadow/z-index) from `styles/tokens.css` remain live and are not duplicated in the coral file.

**Verdict**: there is a design-token architecture, and it's not bad (spacing/typography/radius/motion tokens are genuinely well-structured, RTL-safe via logical properties per `presentation/ui/tokens.css:9`). But there are **two authoritative-looking, contradictory color palettes in the repo**, one of which is fully dead but not deleted, which will actively mislead the next engineer or agent who edits `styles/themes.css` expecting it to affect the live app. This is the concrete manifestation of the "no design system" gap named in `docs/00-discovery.md` §7.2 — the problem isn't the absence of tokens, it's the absence of a single source of truth and the process/tooling (Storybook, a token-lint rule) that would have caught the duplication.

Additional gaps vs. charter 17.2:
- **No "sensory calm" theme** — `grep -rni "sensory.calm" app/src` → zero hits. Only light/dark/high-contrast exist.
- **No dyslexia-friendly font option** — `grep -rni "dyslexi\|opendyslexic" app/src` → zero hits. The only font is Google's "Assistant" (`index.html:12-15`), loaded from `fonts.googleapis.com` with an offline fallback to `system-ui` (comment: "Offline-first: נופל חיננית ל-system-ui כשאין רשת"). Assistant is a reasonable general-purpose Hebrew webfont (rounded, high x-height) but there is no alternate dyslexia-optimized Hebrew face (e.g., a Hebrew-adapted OpenDyslexic-style font) offered as a toggle. **Secondary finding**: loading a font from a third-party Google CDN on every session sends the user's IP to Google before any consent/analytics-opt-in gate — worth a privacy note for Stage D, low severity, easy self-host fix.
- **No Storybook / component library** — confirmed via `package.json` (no `@storybook/*` dependency) and no `.storybook/` directory.

#### 17.3.2 Proposed token architecture

1. **Consolidate to one color source of truth.** Delete the dead color blocks from `styles/themes.css`, keep `presentation/ui/tokens.css` as the single color authority (it's the one that's actually live and the one carrying the current coral brand), and move its non-color tokens (radius `--co-r-*`, shadow `--co-shadow*`) into `styles/tokens.css` so there is exactly one file per concern: `tokens.css` = structural tokens (spacing/typography/radius/shadow/motion/z-index/touch-target), one theme file = color, per theme. Rename the surviving theme file to something that won't be confused with the deleted one (e.g. `colors.css`).
2. **Fix the two confirmed contrast Fails as part of this consolidation**, not separately: (a) either darken light-mode `--cl-primary` to reach 4.5:1 against white (e.g. `#CE5132`, the existing `--cl-primary-dk` token, already defined at `presentation/ui/tokens.css:29`, measures well above 4.5:1 — consider making *that* the default text/icon-on-white color and reserving the lighter coral for large surfaces/backgrounds only), or (b) keep coral for backgrounds and switch its paired text color to dark ink as already done correctly in dark/high-contrast modes (`--cl-on-primary: #2B221C` pattern). Either fix is S complexity and directly closes the waived axe rule.
3. **Add a fourth theme: sensory-calm.** Lower saturation across category tints, slower/removed hover transforms (`.cell:hover { transform: translateY(-2px) }` in `board.css:77-79` — a sensory-calm theme should suppress this even without `prefers-reduced-motion`, since "calm" and "reduced motion" are related but not identical user needs), muted rather than bright category colors, and a softer border-radius/shadow scale.
4. **Add a dyslexia-friendly font toggle.** Bundle a Hebrew-supporting dyslexia-oriented face self-hosted as a `.woff2` (covered by the existing Workbox `woff2` glob pattern at `vite.config.ts:33`, so it gets offline precache for free) — expose as a settings toggle alongside the existing `highContrast` boolean in `AccessSettings`.
5. **Self-host "Assistant"** instead of the Google Fonts CDN link, closing the privacy leak and the (currently theoretical) offline-first gap in one move — same Workbox precache mechanism applies.
6. **Component library (Storybook) plan**: given the actual component inventory (`CellButton`, `Modal`, `Toggle`, `Slider`, `BoardToolbar`, `SentenceBar`, category/board chips), a Storybook instance with a11y addon (`@storybook/addon-a11y`, wraps axe-core — the project already depends on `@axe-core/playwright`, so the team has the relevant a11y-testing muscle already) would (a) make the token duplication in §17.3.1 impossible to miss visually, (b) give design/product a place to review the coral rebrand and the four themes side-by-side, (c) let QA add stories-as-test-fixtures. **M complexity**, no urgent blocker, but should land before the next visual rebrand round (there's already been at least two — "Claude Design 2026" then "F6 coral" — without a component-catalog safety net).
7. **Touch-target policy**: keep 92px as the default (already exceeds the charter's 88×88pt ideal) and the 44px WCAG floor as the *explicit minimum a caregiver can dial down to*, but **fix the mobile media-query bug** (§17.2 target-size row) so that floor is only ever reached by deliberate caregiver choice, never as an unannounced viewport-triggered override. This is the single highest-leverage a11y fix in this whole document — see C-04.

### 17.4 Key flows review

For each flow: **current state** (verified from code) → **gaps** → **redesign direction**.

**1. Onboarding (family / therapist / school branches)**
- *Current*: `AuthGatePage.tsx` offers exactly two states — login or register (`AuthGatePage.tsx:12,21-33`) — with no role selection, no "who are you" branching, and no in-app tutorial (`grep` for onboarding/welcome/tutorial → zero hits, §17.1 heuristic 10). `QuickStartWizard.tsx` (3 steps: name → template + grid size → confirm) creates a *board profile*, not a *product* onboarding.
- *Gaps*: charter 17.3 explicitly asks for "family / therapist / school branches" — none exist. A brand-new SLP signing up to try the product with a client gets the identical flow as a parent. No first-run tour of the play/builder/lock model, which is a genuinely non-obvious three-mode mental model (child-locked / adult-unlocked / builder) for a first-time user.
- *Redesign*: add a persona-select step immediately after registration ("אני הורה / קלינאי/ת תקשורת / איש/אשת חינוך" — parent / SLP / educator), branching into persona-appropriate first boards (SLP → template gallery emphasizing clinical categories; school → classroom-multi-child hint even before Part 20's multi-tenant model ships) and a 3-screen "here's how locking works" tour gated behind a "skip" (never force a tour on a repeat user — respects heuristic 8, minimalism).

**2. Board builder (drag-drop, categories, folders, quick add from camera)**
- *Current*: `BuilderView.tsx` (623 LOC) implements drag-drop, undo, multi-select (per `docs/01-audit.md` B-05). Cell image capture uses a plain `<input type="file" accept="image/*" capture="environment">` (`CellEditor.tsx:543-547` per `docs/04-media-ai.md` §21.1) which opens the **native OS camera app**, not an in-app camera — no grid overlay, no face guide, as the charter 17.3 asks for.
- *Gaps*: no in-app camera (confirmed absent, not just "basic" — Stage E §21.1 already covers the pipeline gaps; this doc adds the UX framing: the charter's grid/face-guide ask is fully unbuilt), no "quick add from camera" one-tap flow distinct from the general cell-editor image picker, large monolithic editor components create an implicit UX risk (a component this size is hard to keep visually simple, per heuristic 8 concern above — `[TBD]` without a screenshot).
- *Redesign*: build the in-app camera per Stage E's spec (§21.1: `getUserMedia` + rule-of-thirds grid + on-device face-detection-biased crop) and expose it as a **first-class "quick add" entry point** from the board grid itself (long-press an empty cell → camera directly), not buried inside the full cell editor — this is the highest-frequency action for a parent building a board ("add a photo of grandma's new dog") and currently requires: open builder → add cell → open cell editor → tap image field → OS camera → back to editor → save. Target: 2 taps.

**3. Board play mode (fullscreen, lock to child, guided access hint on iOS)**
- *Current*: play mode is the default/home view; lock/unlock is a **long-press** gesture on a lock affordance (`AdultBar.tsx`, verified via e2e `lock-unlock.spec.ts`: short Enter-press does not unlock, long-press does), backed by PIN in `domain/access.ts` (`DEFAULT_PIN = '1234'`, explicitly documented as "MVP gate, not cryptographic security" — `access.ts:5`). No "fullscreen" API call was found (`grep -rn "requestFullscreen" app/src` → **zero hits** `[Verify — re-check, low confidence single grep]`), and no Guided Access hint/instructions screen exists for iOS.
- *Gaps*: default PIN `1234` shipped as a literal constant is a real, if low-severity (explicitly "not a security boundary"), risk if a family never changes it — a curious sibling can guess it in a few tries; the code is honest about this ("PIN הוא שער MVP מקומי — לא אמצעי אבטחה קריפטוגרפי") which is the right engineering judgment call, but the *product* should surface a "change your PIN" nudge in onboarding rather than leaving `1234` live indefinitely. No Guided Access (iOS) or App Pinning (Android) instructional UI — the charter explicitly calls this out as a Part 17/18 deliverable and it's fully unbuilt.
- *Redesign*: (a) force a first-run PIN change or explicitly accept the default with a warning; (b) add a one-time, dismissible "for extra safety, enable Guided Access" card on iOS Safari-added-to-homescreen sessions with a link to Apple's instructions (native `requestFullscreen`/App Pinning equivalent hint on Android); (c) verify/add a `document.documentElement.requestFullscreen()` call on entering play mode where supported, to minimize accidental exits to the OS home screen mid-session.

**4. Family member capture flow (photo + name + relationship + voice recording)**
- *Current*: photo capture per flow #2 above. Voice recording is real and already prioritized correctly at playback (`speakCell()` checks `cell.audioId` before TTS — `docs/04-media-ai.md` §21.2, confirmed in Stage E), but is **one recording per cell**, not a reusable named "voice" entity with relationship tagging — there is no "אמא / אבא / סבתא רותי" voice bank UI, exactly as Stage E's E-13 finding states. This doc's UX framing: the *missing* schema (Stage E's proposed `VoiceClip` entity) is also the *missing* flow — a parent cannot today go through "capture family member" as one guided sequence (photo → name → relationship → record voice → attach to relevant tiles); they'd have to hand-build it cell-by-cell.
- *Gaps*: no waveform/trim/re-record UX (Stage E E-13), no dedicated "family member" onboarding flow at all — it's assembled implicitly from generic cell-editing primitives.
- *Redesign*: a dedicated "הוסף בן/בת משפחה" (add family member) wizard: photo (in-app camera per flow #2) → name + relationship chip-picker → voice recording with waveform/trim (per Stage E spec) → auto-suggest which existing tiles ("אמא", "אבא", pronoun tiles) should use this voice. This is the single most emotionally significant flow in the product (a grandparent's voice on a nonverbal grandchild's board) and currently gets the least dedicated UX of any flow in the app.

**5. Sharing a board with a therapist**
- *Current*: `ShareInvitePanel.tsx` — a 6-digit code, 48-hour TTL, role picker (`clinician`/etc. via `ChildAccessRole`), clipboard-copy delivery (`ShareInvitePanel.tsx:1-2,17`). Functional and minimal.

### 17.5 Motion & haptics guidelines

Verified baseline: transitions are short (`--ease-fast: 0.08s`, `--ease: 0.15s`, `--ease-slow: 0.25s`, `styles/tokens.css:97-99`) and `prefers-reduced-motion` is respected in 8/13 stylesheets (§17.2). No haptics (`navigator.vibrate`) calls were found anywhere in the codebase (`grep -rn "vibrate" app/src` → zero hits) — this is a **greenfield gap**, not a regression, since haptics are a real, cheap (no native app needed — `navigator.vibrate` works in PWA contexts on Android Chrome, though not iOS Safari) reinforcement signal for AAC users who benefit from multi-sensory confirmation of a tap.

**Guidelines to adopt:**
1. Every tile activation gets a **short (~15–20ms) haptic pulse** on Android/Chrome via `navigator.vibrate(15)`, feature-detected and silently absent on iOS Safari (no polyfill exists; do not attempt one). Gate behind a settings toggle default-on, since some sensory-sensitive users will want it off (ties into the sensory-calm theme, §17.3.2).
2. No animation should communicate meaning **exclusively** through motion — the reduced-motion pattern already used for the lock-fill (visual suppressed, functional timer intact, `mvpUx.css:56-58`) is the model to replicate everywhere: **motion is decoration, never the only channel.**
3. Cap all interactive transform animations (hover-lift, active-press) to ≤200ms and ≤4px of displacement — current values (`.cell:hover { translateY(-2px) }`, `.cell:active { translateY(1px) scale(0.98) }`, `board.css:77-83`) already comply; keep this as an explicit budget for new components rather than an accident of the current values.
4. No auto-playing looping animations anywhere (none found — good, keep it that way; this is the single most common sensory-overload trigger the charter's "no surprise animations" principle warns against).
5. Sensory-calm theme (§17.3.2) should default haptics off and further compress the motion budget (transforms → opacity-only feedback).

### 17.6 Copy tone guide — עברית פשוטה (Hebrew, simple language)

Verified sample of in-app copy (from files read this session): "דבר" (speak), "מחק מילה" (delete word), "נקה הכל" (clear all), "שמור ביטוי" (save phrase), "האם למחוק את כל התמונות מהענן? התמונות יישמרו במכשיר זה." (delete-from-cloud confirm — direct, explains the consequence, tells the user what *stays*, not just what's removed), "שגיאה ביצירת קוד שיתוף — נסה שנית" (share-code error). This is already a good baseline: short imperative verbs on buttons, consequence-explaining confirms, no jargon, no English loanwords where a native Hebrew term exists.

**Guidelines to formalize (currently implicit, should become a documented style guide):**
1. **Button labels: verb, not noun.** "דבר" not "השמעה" (an SLP-facing word), "מחק" not "מחיקה". Already the dominant pattern — codify it so new features don't drift.
2. **Errors state consequence + next step, in that order.** The delete-from-cloud confirm above is the model: what will happen → what will remain → (implicitly) how to proceed. Extend this pattern explicitly to every future confirm dialog.
3. **No transliterated English where a common Hebrew word exists** — verified the sampled copy has none; this is a real house style already, just undocumented. Where a technical term has no natural Hebrew equivalent (e.g. "PIN קוד"), keep the loanword but always pair it with its function ("קוד מטפל" — caregiver code — rather than bare "PIN").
4. **Sentence length ≤ ~8 words for anything shown during play mode** (a nonverbal child or a stressed parent reading over their shoulder should parse it in one glance); settings/builder screens (adult-only, per the lock model) can run slightly longer since they're read outside the time-pressured play context.
5. **Numerals for anything countable** ("48 שעות", "6 ספרות") rather than spelled-out numbers — matches the existing `ShareInvitePanel` pattern and is generally easier for a fast scan.

3. **Phase 3: native modules only where a JS-only Expo path is insufficient** — voice recording quality/background limitations and Guided-Access-equivalent lockdown (§18.3) are the two candidates the charter names, and this review agrees they're the right two: everything else (TTS, symbols, sync) has a viable Expo-ecosystem or bridged solution.

### 18.2 iOS PWA limitations — AAC-specific impact

|---|---|---|
| **Audio autoplay policy** — iOS Safari requires `<audio>.play()` (or Web Audio start) to be tightly coupled to a user gesture; a chain of `await`s between the tap and the play call can break that coupling. | **Directly hits the core "one-tap-to-speak" invariant.** `HybridTtsService.speak()` (`hybridTtsService.ts:17-49`) does: cancel-previous → **await** cache-key build → **await** cache lookup → (on cache miss) **await** network synthesis → **await** cache-write → *then* play. - **Push notifications**: none exist today (confirmed, Stage A). Not reachable at all on iOS Safari PWA without native wrapping (Apple only added iOS 16.4+ Web Push for PWAs, and even then it requires the user to have already added the app to the home screen — `[Assumption/TBD, verify current iOS Web Push support + the ≥16.4 version gate before relying on it]`). ### 19.2 Switch-access / scanning support — verified state and plan

**Correction to task-brief framing (repeated from the header, expanded here because Part 19 is where it matters most):** switch access is not a "plan to build" — it is a **working, if unpolished, feature today.**

**What exists, verified:**
- `domain/scanning/scanEngine.ts` — pure state machine, two modes: `linear` (cell-by-cell) and `row-column` (scan rows, dive into cells within a selected row) — this **is** classic 2-tier row-column scanning, the standard AAC pattern for reducing scan time on larger grids.
- `services/access/useScanning.ts` wires it to: an auto-advance timer (`speedMs`, 0 = manual), an `onSelect`/`onHighlight` callback pair, and a `window`-level `keydown` listener mapping **Enter → select** always, **Space → select in auto mode / advance in manual mode** (`useScanning.ts:78-98`). This maps directly onto the two standard switch-access configurations:
  - **1-switch auto-scan**: `scanSpeedMs > 0`, a single switch mapped to one key (Space or Enter) triggers selection while the scanner cycles automatically and speaks each highlighted cell if `scanAuditory` is on (`useScanning.ts:67-73`, reads `cell.nikud ?? cell.label` via TTS at `App.tsx:229-232`).
  - **2-switch manual-scan**: `scanSpeedMs = 0` (`accessSettings.scanSpeedMs`, exposed presumably via the settings panel `[Verify UI control exists — AccessSettingsPanel.tsx was read in part; the scanning section specifically was not fully read this session]`), one switch (Space) advances, a second switch (Enter) selects.
- Wired into `App.tsx:212-233`: scanning is active only in child/play mode, never in builder or settings (`!builderMode && openPanel !== 'settings'`) — a correct scoping decision (scanning through builder's dense UI would be a much harder design problem, correctly deferred).
- `BoardView.tsx` accepts `scanIndices` and applies `.cell--scan-highlight` (`board.css`-adjacent, defined in `features.css:46`) to the currently-highlighted cell(s).
- Row-column mode's row-count is computed from the visible cell count and the board's actual `grid.cols` (`useScanning.ts:47-50`, `App.tsx:224`), so it adapts to whatever grid size the child's board is currently using — a real integration, not a hardcoded demo.

**What's genuinely missing (accurate framing of the actual gap, replacing "it's a stub"):**
- **No on-screen switch-simulation UI** — the mechanism assumes an external switch is mapped to a keyboard key at the OS/hardware level (the standard real-world setup for USB/Bluetooth AAC switches), which is appropriate, but there's no in-app way to *test* scanning without a real switch or keyboard, making it hard for a parent/therapist to configure confidently.
- **No visual "scan speed preview"** before committing to a speed setting.
- **No auditory scanning voice-interruption handling verified** — if a cell's TTS announcement (`onHighlight`) is still speaking when the scanner auto-advances to the next cell, does the previous utterance get cancelled? `HybridTtsService.cancel()` exists and is called at the start of every new `speak()` (`hybridTtsService.ts:21` comment "A6: עצור השמעה קודמת") — so **yes, this is handled correctly**, verified.
- **Dwell-activation and switch-scanning don't compose** (§17.2 2.1.1 row) — a family using both a switch and a touchscreen for different children, or a child transitioning from touch to switch access, gets two independently-configured systems rather than one coherent access model.
- **No eye-gaze/head-tracking integration** (§19.3) reuses the same `onSelect`/`highlightedIndices` seam scanning already established — this is good news: the seam for a third access method already exists in the architecture.

**Plan for closing the real gaps** (not "build scanning" — it's built):
1. Add an in-app "test your switch" calibration screen before scanning is enabled, using the existing `useScanning` hook against a demo grid.
2. Expose `scanSpeedMs`, `scanMode`, `scanAuditory` fully in `AccessSettingsPanel` if not already complete `[Verify]`, with a live preview.
3. Document the two switch-hardware integration patterns (1-switch via a single Bluetooth-HID switch mapped to Space, 2-switch via two switches mapped to Space/Enter) in user-facing help copy — this is a documentation/support gap, not a code gap.
4. Extend `ScanConfig`/`ScanState` (already designed for extension — `mode: ScanMode` is a clean discriminated union) with a third mode for eye-gaze dwell-select (§19.3) rather than building a parallel system.

### 19.3 Eye-gaze / head-tracking compatibility roadmap

No eye-gaze/head-tracking code exists (`grep -rni "gaze\|eyetrack\|tobii" app/src` → zero hits — greenfield, expected). The good news, established in §19.2, is that the **architectural seam already exists**: both dwell-activation (`useDwellActivation`, pointer-based) and switch-scanning (`useScanning`, keyboard-based) are already abstracted as "a stream of activation events feeding a single `onSelect`/`onActivate` callback." Eye-gaze/head-tracking hardware (Tobii, PCEye, or phone-camera-based solutions like the open-source "Eye Gaze" approaches) typically either (a) emit synthetic pointer/mouse events (in which case `useDwellActivation`'s existing pointer-event handlers work with **zero code change** — dwell-on-gaze is functionally identical to dwell-on-touch from the app's point of view), or (b) require a dedicated integration.

**Roadmap:**
1. **Phase 1 (near-term, near-zero cost)**: verify `useDwellActivation`'s pointer handlers work with any gaze-tracking software that emits synthetic `PointerEvent`s (many Windows/Tobii gaze-mouse products do exactly this) — `[TBD, needs a real device test, but the code path should already support it]`.
2. **Phase 2**: for camera-based/WebGazer-style browser-native gaze tracking (no special hardware, runs on-device via webcam + a WASM/JS gaze-estimation model — consistent with this product's existing "keep sensitive processing on-device" posture from Stage E's photo-moderation design), integrate as a new event source feeding the same `onActivate` callback shape `useDwellActivation` already exposes.
3. **Phase 3**: head-tracking (camera-based head-pose-as-cursor, common lower-cost motor-access alternative to eye-gaze) follows the identical integration shape.

This is explicitly a roadmap, not a current capability — no code toward any of this exists today; the value of this section is establishing that the existing architecture does not need to be reworked to support it later, which is a meaningful finding in itself (avoids a costly future refactor).
