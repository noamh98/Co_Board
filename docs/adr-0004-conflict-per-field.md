# ADR-0004 — מודל קונפליקט per-field (מול LWW) ⚠️ הצעה (Phase 5)

- **סטטוס:** מוצע (לא מיושם). תלוי בכיוון-מוצר — לפתוח רק אם נדרש.
- **הקשר:** הסנכרון משתמש כיום ב-Last-Write-Wins ברמת הישות (`domain/sync.ts`,
  `services/sync/syncEngine.ts`). עריכה במקביל משני מכשירים → המנצח דורס את כל
  שדות המפסיד (גיבוי המפסיד נשמר ב-versions, אך לא ממוזג).
- **בעיה:** הורה משנה תווית במכשיר A בעוד קלינאית מוסיפה תא במכשיר B — אחד השינויים אובד
  (משוחזר רק ידנית מ-versions).

## אפשרויות
1. **להשאיר LWW** (המצב): פשוט, דטרמיניסטי, מגובה. מתאים אם עריכה-במקביל נדירה.
2. **CRDT/merge per-field**: למזג ברמת התא/השדה (placements, cells). מורכב — דורש
   מבנה-נתונים ממוזג ובדיקות נרחבות.
3. **per-field timestamps**: לשמור updatedAt פר-שדה ולמזג שדה-שדה. ביניים.

## המלצה
לדחות עד שיש עדות לאובדן-עריכות בשטח. אם כן — אפשרות 3 (per-field timestamps) היא
האיזון Ponytail: ערך גבוה, מורכבות בינונית. **קבלה:** טסטי-מיזוג לעריכה-במקביל;
תאימות-לאחור עם רשומות LWW קיימות.
