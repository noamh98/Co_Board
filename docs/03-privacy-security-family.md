## Part 7 – Security Assessment

### 7.1 Authentication

**Password storage.** Co_Board uses Firebase Authentication's email/password provider exclusively for server-side account credentials (`app/src/services/sync/firebaseProvider.ts:17-24`, `app/src/services/sync/firebaseAuth.ts`). Password hashing is entirely Google-managed — Firebase Auth stores credentials using a scrypt-derived algorithm on Google's infrastructure; the app never sees, stores, or logs a raw password beyond the in-memory form field. **This is a reasonable, industry-standard baseline and there is nothing to remediate here** — the task brief's "Argon2id / bcrypt cost" ask doesn't apply because this isn't a self-hosted auth stack. `[Assumption]` Firebase's scrypt parameters are not tunable by the app and are not publicly documented in detail by Google; treat this as an accepted platform dependency, not a gap.

**Email verification + admin approval (the actual gate).** Every new account goes through two sequential gates before it can read/write any content:
1. `sendVerificationEmail()` is fired immediately after signup (`App.tsx:345`, `firebaseAuth.ts:114-118`) — standard Firebase email-link verification.
2. A human with the `admin` custom claim must call `approveUser` (`functions/src/approveUser.ts`) to flip `status` to `approved` and set the `approved: true` custom claim.

Firestore rules enforce **both** conditions together via `isApproved()` (`docs/firestore.rules:14-18`): `request.auth.token.email_verified == true && request.auth.token.approved == true`. This is verified end-to-end by `functions/test/rules.test.ts` ("unapproved user cannot read children — even the owner"). This two-gate design is deliberate and already flagged at the business-process level as B-03 (admin approval bottleneck) in Stage B — this document does not re-litigate the scaling problem, only confirms the security *mechanism* is sound: there is no client-writable path to self-approve (`docs/firestore.rules:41-47` makes `status` immutable from the client after creation).

**MFA readiness: none today, confirmed by code.** `grep -rn "MFA\|multiFactor" app/src` returns zero matches. Firebase Auth supports multi-factor (SMS/TOTP) natively, so adding it is a config + UI change, not a rearchitecture — but nothing in the current codebase wires it up, for either parent or therapist accounts. Given clinicians/school staff will hold `clinician`/`staff` `childAccess` grants across potentially many families (Part 9), the absence of MFA is a materially higher risk for those roles than for a single parent account.

**Session/token handling.** No custom session code exists — the app relies entirely on the Firebase Auth SDK's default ID-token lifecycle (short-lived ID token, auto-refreshed via a long-lived refresh token, `onAuthStateChanged` listener at `firebaseAuth.ts:195-200`). `[Assumption]` this means Google's default token TTLs and revocation semantics apply as-is; there is no app-level token rotation policy to audit because none was custom-built. This is appropriate for the current scale — flagging only so a future audit doesn't assume custom JWT handling exists.

**Account recovery — and the AAC-specific angle.** This is a genuine, concrete gap: **there is no "forgot password" flow anywhere in the codebase.** `LoginPanel.tsx` (full file read) has no reset-password link or button; `grep -rn "sendPasswordResetEmail" app/src` returns zero matches. Firebase Auth supports `sendPasswordResetEmail()` natively — it is simply never called. Today, a parent who forgets their cloud-sync password has no self-service recovery path in-app; recovery would require either the Firebase console (admin-only, not scalable) or building the feature.

The AAC-specific mitigating factor, verified from the architecture: **cloud login is only required for cross-device sync and sharing, never for local use.** The app is offline-first by design (`docs/00-discovery.md` §2, confirmed by `syncEngine.ts` no-op behavior when `syncEnabled=false`) — a child's board, TTS, and play mode work fully without any account at all. So a forgotten *cloud* password does not, by itself, lock a child out of their voice on the device they already use day-to-day. It **does** block: multi-device sync, therapist/school sharing, and cloud backup recovery after device loss — i.e., exactly the scenarios where account recovery matters most for a family. **Net assessment: the missing password-reset flow is a real gap that should be closed before self-serve signup (referenced in B-03's remediation) — not because a child loses their voice, but because a family locked out of their account loses cross-device continuity and cloud backup precisely when they need it (e.g., after buying a replacement device).**

The **local PIN gate** (`app/src/domain/access.ts`) is a separate, unrelated mechanism and should not be confused with account security: `DEFAULT_PIN = '1234'`, comparison is a plain trimmed string-equality check (`verifyPin`, `access.ts:30-35`), and the code's own comment states it explicitly — *"ה-PIN הוא שער MVP מקומי — לא אמצעי אבטחה קריפטוגרפי"* ("the PIN is a local MVP gate, not a cryptographic security measure"). The PIN is stored in plaintext in IndexedDB (`settingsRepo.ts:15,140-141`, key `caregiverPin`). Per `useAppBootstrap.ts:71-72`'s comment, the currently *shipped* unlock mechanism for adult mode is long-press only — the PIN itself is an opt-in advanced layer not yet the default gate. **This is correctly scoped as a "keep a curious child from wandering into the builder" UX gate, not a security boundary against a motivated adult attacker with physical device access** — no remediation needed here as long as product/marketing never claims it is a security feature. There is also no "forgot PIN" flow, but the consequence of a forgotten PIN is much softer than a forgotten cloud password: no data is destroyed, the fix is a factory-reset-free wait for a developer/support hint or (once built) a cloud-authenticated reset. `[Assumption]` recommend documenting a supported PIN-reset path (e.g., "sign in with your cloud account to reset the local PIN") rather than leaving it fully unaddressed once self-serve support exists.

**Child accounts under parental control.** There is, correctly, no such thing as a "child account" in Firebase Auth — children never sign in. The domain `Role` enum (`app/src/domain/models.ts:138`) includes `'child'` only as a *local, device-side* mode (with `ROLE_CAN_EDIT.child = false`, `models.ts:142`), not a cloud identity. This is the right posture for COPPA-adjacent design: no credential, no PII collection, no account for the minor at all — the parent/guardian's Firebase account is the only cloud identity in the system today. This is a **strength**, not a gap, and should be preserved explicitly as a design principle when Part 20's family model is built (do not introduce child logins).

### 7.2 Authorization

**Current role model — two parallel enums, not yet unified.** The codebase has two distinct role concepts that a reader could easily conflate:
- **Local `Role`** (`domain/models.ts:138`): `'child' | 'parent' | 'clinician' | 'staff'`, drives `ROLE_CAN_EDIT` (`parent`/`clinician` → edit; `child`/`staff` → view/play only) — used for the on-device adult-mode gate.
- **Cloud `ChildAccessRole`** (`data/childRepo.ts:34`): `'parent' | 'clinician' | 'staff'` — the role recorded in `childAccess/{childId}/members/{uid}` when a share invite is redeemed.

These overlap but are not the same axis (one is per-device UI mode, the other is a cloud ACL entry), and neither has a **co-parent** or **admin** distinction, and neither models a school-classroom scope. Part 20 below proposes the unification.

**What the Firestore rules actually enforce (`docs/firestore.rules`, walked line-by-line):**

| Path | Rule | Effective access |
|---|---|---|
| `users/{uid}` (line 37-48) | `read`: self or admin. `create`: self, `status` must start `'pending'`. `update`: self, but `request.resource.data.status == resource.data.status` — **status is immutable from the client.** | Prevents client-side self-approval — verified by test `"user cannot change own status (immutable from client)"`. |
| `users/{uid}/{collection}/{document}` (line 52-54) | `read, write`: `isOwnUid(uid) && isApproved()`. **No `hasChildAccess` branch.** | Covers the actual AAC content — `board`, `profile`, `settings` subcollections (per `firebaseProvider.ts:59,78`, `entityTypes = ['board','profile','settings']`). **Only the owner can ever read/write these documents — a `childAccess` grant does not extend to them.** See §7.2.1 below — this is the single most important authorization finding in this document. |
| `users/{uid}/children/{childId}` (line 57-62) | `read, write`: `isApproved() && (isOwnUid(uid) \|\| hasChildAccess(childId))`. | Grants access to the child **metadata** doc (name, age, preferences, homeBoardId, archivedAt — per `childRepo.ChildRecord`) to owner + any accepted invitee. Does **not** cover boards. |
| `childAccess/{childId}/members/{memberId}` (line 65-73) | `read`: self or any existing member. `write`: only `ownsChildDoc(childId)` (i.e., the actual owner). | Correctly blocks privilege escalation — a `clinician` member cannot grant themselves or anyone else additional access. Verified by test `"member (clinician) cannot write childAccess"`. |
| `shareInvites/{code}` (line 78-86) | `read`: owner only. `create`: approved owner, `ownerUid` must equal `request.auth.uid`. `delete`: owner only. **No client `update` rule** — the `used: true` flag is written exclusively by the `acceptInvite` Cloud Function via the Admin SDK, which bypasses rules entirely. | An invitee never reads the invite document directly — good design, prevents enumeration/guessing feedback via direct Firestore reads. |
| `users/{uid}` `list` (line 89-91) | `admin` only. | Used by the admin approval dashboard to enumerate pending signups. |

**7.2.1 — Critical finding: the child-sharing feature is not wired end-to-end.** Tracing rules, Cloud Function, and client UI together shows each layer looks correct in isolation, but they don't compose into a working feature:

1. **Rules gap.** `hasChildAccess(childId)` only appears in the rule for the `children/{childId}` metadata document (line 57-62). The board/profile/settings documents that actually constitute an AAC board (labels, tiles, nikud, layout) live at `users/{ownerUid}/board/{boardId}` etc. and are covered only by the owner-only wildcard rule (line 52-54). **A clinician or school-staff member who redeems a share invite gains Firestore read/write on the child's name/age/preferences — but never on the child's actual board content.** No code anywhere queries `users/{ownerUid}/board/...` using a non-owner's session (`firebaseProvider.pull()`, the only reader of that path, always uses `auth.currentUser.uid`, `firebaseProvider.ts:70-72` — it can only read the *signed-in* user's own tree).
2. **Storage is more permissive, but decryption still fails across accounts.** `firebase/storage.rules:31-35` correctly checks `hasChildAccess(profileId)` for media — so, unlike Firestore, a clinician's Storage read of a shared child's encrypted photo would be *permitted*. But it can't be *used*: media is encrypted client-side with a **per-device, non-extractable AES key** (`data/keyStore.ts:24-42`, `extractable:false`, generated once per device, never exported, no key-exchange mechanism anywhere in the codebase). `encryptBlob()` (`services/sync/crypto.ts:102-132`) wraps a random per-file content-encryption-key (CEK) using *that specific device's* key. A different device — even a second device of the **same** parent, let alone a clinician's own device — has a different device key and **cannot unwrap the CEK**; `decryptBlob()` (`crypto.ts:139-186`) throws inside `unwrapKey`/`decrypt`, is caught, and silently returns `null` (`crypto.ts:183-185`). **Cross-device decryption of synced media is broken by construction today, for anyone** — not just shared clinicians, but a two-device single-parent household. This is distinct from and more severe than the E2EE-vs-moderation tradeoff already flagged as E-11/B-19: it's not a policy tradeoff, it's a functional defect — there is no mechanism to escrow/re-wrap the CEK for a second legitimate device or account. (Boards/profiles/settings sync as plaintext Firestore documents and are unaffected — only media goes through `encryptBlob`/`decryptBlob`.)
3. **The invite-redemption UI cannot accept the codes the backend actually generates — a concrete, reproducible bug.** `data/childRepo.ts:153-157` generates share codes as **128-bit random hex** (32 hex characters) via `crypto.getRandomValues(new Uint8Array(16))` — the comment explicitly says this "replaces the 6-digit `Math.random()`" scheme as a security hardening (`childRepo.ts:153`). But `presentation/portal/AcceptInviteScreen.tsx:19` validates input with `trimmed.length !== 6 || !/^\d{6}$/.test(trimmed)`, and the `<input>` has `maxLength={6}` + `inputMode="numeric"` (`AcceptInviteScreen.tsx:59`) — **a 6-digit-numeric-only field that can never hold a real 32-character hex code.** `ShareInvitePanel.tsx`'s own header comment (line 2) still says *"קוד 6 ספרות"* ("6-digit code"), and `functions/test/rules.test.ts:27` seeds a test invite with a 16-hex-char code — none of these three surfaces agree with each other or the real format. **Net effect: no user can actually redeem a real share invite through the shipped UI today** — the 6-digit → 128-bit-hex hardening was applied to the generator but never propagated to the UI/copy that consumes it. A shippable, verifiable bug, not a design tradeoff.

**Assessment of the net effect (good news, bad news):** The good news, directly answering the Part 9 "can a tenant accidentally access another tenant's data" question in the narrow child-sharing context: **no** — the failure mode here is *under*-permissive, not over-permissive. Nothing in this trio of bugs lets an unauthorized party read data they shouldn't; it just means the *authorized* collaboration path (parent invites a therapist) does not currently deliver working collaboration. The bad news: **this is exactly the feature Part 20 (family & multi-role account model) is asking to be designed** — it turns out the groundwork exists (rules, Cloud Function, invite entropy) but is disconnected at three separate points, so Part 20 is a *repair-and-extend* job, not pure greenfield.

**IDOR analysis, summary.** Walking every rule above for "can uid X read/write a document scoped to uid Y or child C without an explicit grant": no. `isOwnUid`, `hasChildAccess`, and `ownsChildDoc` are all correctly scoped to `request.auth.uid`, and the one gap found (§7.2.1) is a missing *grant*, not an accidental over-grant. Default-deny holds: the storage rules' final `match /{allPaths=**} { allow read, write: if false; }` (`storage.rules:48-49`) and the absence of any Firestore catch-all `allow` mean anything not explicitly matched is denied. **No cross-tenant IDOR found in the rules as written.**

**Invite-code security (replay/expiry/entropy), independent of the UI bug above:**
- **Entropy:** 128 bits of `crypto.getRandomValues` output (`childRepo.ts:155`) — cryptographically unguessable; brute force is not a realistic threat.
- **Replay:** `acceptInvite` (`functions/src/acceptInvite.ts:51-91`) runs inside a single Firestore transaction that reads the invite, checks `used !== true` and `expiresAt >= now`, grants `childAccess`, and sets `used: true` atomically — a second redemption attempt on the same code fails the `used` check inside the same transaction. **Correctly prevents double-redemption/replay races.**
- **Expiry:** `TTL_48H` (`childRepo.ts:151`) enforced server-side in the transaction, not just client-side — cannot be bypassed by a stale client.
- **Role integrity:** `ALLOWED_ROLES` allowlist (`acceptInvite.ts:28`) re-validates the role from the *stored* invite document rather than trusting anything from the request — defense-in-depth against a tampered client payload.

This is a genuinely well-built server-side mechanism; the only defect is that the UI in front of it doesn't speak its language (§7.2.1.3).

### 7.3 API security

This section deliberately does not re-derive rate-limiting or file-upload findings already covered in Stage B/E (see B-23, E-01, E-02, E-06, E-07 for those) — it cross-references them and focuses on what those documents didn't cover.

- **Injection/XSS:** React's default JSX escaping covers the UI rendering surface; no `dangerouslySetInnerHTML` was found during this review's targeted greps (`grep -rn dangerouslySetInnerHTML app/src` → no hits encountered in files read). Not exhaustively re-audited here — recommend a dedicated grep-based sweep in Stage F/G CI (`eslint-plugin-react` has a rule for this that should be enabled if not already).
- **CSRF:** Not applicable in the traditional cookie-session sense — all state-changing calls go through Firebase's `httpsCallable` SDK, which attaches the caller's Firebase ID token as a bearer credential per-request rather than relying on ambient cookies, so classic CSRF (forged cross-site form POST riding on a cookie) does not apply to the Cloud Functions. `[Assumption]` Firebase Hosting itself does not set any auth cookie for the app shell.
- **SSRF:** The three server-side outbound calls (`ttsProxy.ts` → `texttospeech.googleapis.com`, `aiBoard.ts` → `generativelanguage.googleapis.com`, and none from `acceptInvite`/`approveUser`) all hit **hardcoded** URL constants (`ttsProxy.ts:18`, `aiBoard.ts:18-19`) — no user input ever constructs or influences the target host, so classic SSRF (attacker redirecting a server-side fetch to an internal address) is not possible via these functions. Low residual risk, well-bounded.
- **Input validation per function:** `ttsProxy` — text length cap (400), voice allowlist, type checks (`ttsProxy.ts:48-57`). `aiBoard` — topic length cap (300), count clamp (1-64), type checks (`aiBoard.ts:68-73`); **topic is interpolated into the Gemini prompt without escaping** — already flagged as E-06 (Medium, low blast radius due to downstream JSON/POS validation) — not re-analyzed here. `acceptInvite` — code type-checked as string, all business logic re-validated server-side inside the transaction (§7.2). `approveUser` — `uid`/`status` type + enum checked (`approveUser.ts:29-31`); note **`approveUser` does not itself re-check `email_verified` before flipping `approved: true`** — an admin could technically approve an unverified email. Low risk in practice (an admin is a trusted human reviewing signups) but worth a one-line defensive check.
- **Rate limiting:** already covered by B-23 (no App Check → per-uid caps are the only cost control) and E-07 (no fleet-wide spend ceiling) — referenced, not re-derived.
- **File upload validation:** already covered by E-02 (EXIF-on-error-path gap) and the Storage rules' `isValidMedia()` size/content-type check (`storage.rules:21-24`) — referenced, not re-derived. Note for completeness: `isValidMedia()` checks size and that `contentType == 'application/octet-stream'` — it does **not** and cannot validate that the *ciphertext* actually decrypts to a well-formed image/audio file server-side (it's encrypted — the server has no way to inspect it), so "malware scan" in the traditional sense (scanning file bytes for known-bad signatures) is structurally inapplicable to this bucket's content, same root cause as the E2EE-vs-moderation tradeoff in E-11.
- **CSP:** `firebase.json:15-41` ships a real CSP (not just a meta-tag afterthought) with `object-src 'none'`, `frame-ancestors 'none'`, `base-uri 'self'`, `default-src 'self'` — solid baseline. **`script-src` and `style-src` both still include `'unsafe-inline'`** (`firebase.json:21`), which meaningfully weakens the CSP's ability to stop injected-script XSS even if a future bug introduced one (an attacker-controlled inline `<script>` would execute). This is the same open item flagged in `docs/00-discovery.md` §5 ("CSP still allows unsafe-inline... spike 3.7 not done") — carried forward here as a Security Sentinel finding because it is a live, unresolved control gap, not closed by any later stage's work found in this review. Remediation: adopt a nonce- or hash-based CSP (Vite supports build-time CSP hash injection via plugins) to drop `unsafe-inline`; Medium effort because inline `<style>` usage (likely from CSS-in-JS or inline `style=` attributes) would need auditing first.
- **No CSP `<meta>` tag exists in `app/index.html`** (confirmed by full file read) — the CSP is delivered exclusively via Firebase Hosting response headers. This is actually the **better** of the two mechanisms (headers can't be stripped by a same-origin script the way a meta tag theoretically could be manipulated pre-paint), so this is not a finding, just a confirmation that the CSP is real and server-enforced, not decorative.

### 7.4 Secrets

- **No secrets found in source.** Targeted greps for API-key-shaped strings (`AIza`, `sk-`, PEM private-key headers, `service_account` JSON) across the repo returned exactly one hit — a UI **placeholder string** `"AIza..."` in `AccessSettingsPanel.tsx:287`, which is cosmetic placeholder text for a deprecated, now-no-op client-side TTS-key input field (`settingsRepo.getTtsApiKey()` is confirmed a no-op purge-only function per Stage E's audit, E-audit reference `settingsRepo.ts:94-103`). No real credential.
- **Google Cloud TTS and Gemini keys** are declared via `defineSecret()` (`ttsProxy.ts:19`, `aiBoard.ts:13`) — Firebase Functions v2's Secret Manager integration. Keys are injected at invocation time server-side only (`GOOGLE_TTS_API_KEY.value()`, `ttsProxy.ts:80`; `GEMINI_API_KEY.value()`, `aiBoard.ts:75`) and never appear in client bundles. Comments in both files explicitly instruct `firebase functions:secrets:set ...` rather than committing values — correct practice, consistently followed.
- **Client Firebase config (`apiKey`, `authDomain`, `projectId`, etc., `services/sync/firebaseAuth.ts:33-40`, `data/firebaseEnv.ts`) is intentionally public** — this is expected and safe by Firebase's own design: the Firebase Web SDK "API key" is a project identifier, not a secret, and the actual access control is enforced by Firestore/Storage security rules and Cloud Function auth checks, all of which are analyzed above. State this explicitly in any future security questionnaire (procurement reviewers sometimes flag a visible `apiKey` in a JS bundle as a "leak" — it isn't, for Firebase specifically).
- **CI secrets separation is correct:** client Firebase config flows through GitHub Actions secrets → Vite env at build time; `GOOGLE_TTS_API_KEY`/`GEMINI_API_KEY` live in Firebase Secret Manager, not GitHub — already noted as correct in `docs/01-audit.md` Part 13, referenced not re-derived.

### 7.5 Infrastructure security

- **App Check absent** — already tracked as B-23 (Medium/High depending on framing). Reiterated here with the Part 7 severity lens: **High**, because without App Check, the per-uid rate limits (120/min TTS, 30/min AI) bound cost-per-compromised-*account*, but do nothing to verify that calls are originating from the real PWA rather than a scripted client presenting a stolen/replayed ID token. Recommend enabling before crossing ~1,000 approved users, consistent with Stage B's own recommendation.
- **Single Firebase project** (B-01) — reiterated here specifically as a *security* concern, not just an ops one: schema/rules changes are validated only by the CI emulator, never rehearsed against a live-shaped staging project, so a rules regression (e.g., an accidental loosening of `hasChildAccess`) would hit production data of real families with no rehearsal step to catch it. Cross-reference B-01/B-15.
- **Functions regions:** `europe-west1` primary + `us-central1` "legacy" dual-deploy for `acceptInvite`/`approveUser` (`functions/src/region.ts`), chosen per a code comment for EU privacy positioning ("Emorli AWS Frankfurt" reference, `ttsProxy.ts:6`). This is a reasonable default for an Israel/EU-facing product but note the **third-party endpoints themselves are not EU-pinned**: Google Cloud TTS's `texttospeech.googleapis.com` and Gemini's `generativelanguage.googleapis.com` are both global endpoints with no documented EU-only routing guarantee in this codebase (already flagged in Stage E's provider table) — the Cloud *Function* runs in `europe-west1`, but the third-party API call it makes may be served from anywhere in Google's global network. Relevant for the GDPR data-residency discussion in Part 8.
- **`index.html` loads Google Fonts directly** (`app/index.html:12-17`, `fonts.googleapis.com`/`fonts.gstatic.com`) on every page load, before any consent/auth gate. This sends the visitor's IP address to Google as a side effect of rendering the login screen — a well-known EU privacy-law friction point (multiple German courts have held that loading Google Fonts from Google's CDN without consent violates GDPR because it transmits IP address to a third party without a legal basis in that specific pattern `[TBD — counsel, jurisdiction-specific]`). Low engineering cost to fix (self-host the two font files, drop the `preconnect`/`stylesheet` links) with an outsized EU-readiness payoff — recommend doing this regardless of the counsel review outcome, since self-hosting a font is strictly simpler than defending its absence.
- **No WAF / no explicit network hardening beyond Firebase's platform defaults** — `[Assumption]` Firebase Hosting's CDN provides baseline DDoS absorption as a platform feature; no additional WAF layer (e.g., Cloud Armor) is configured in any file found in this repo. Not urgent at current scale; revisit alongside App Check.
- **Mobile-specific items (certificate pinning, root/jailbreak detection)** are **N/A today** — there is no native app, only a PWA (confirmed, `docs/00-discovery.md` §2, §7.2). Flagged here only as a forward dependency for Part 18's native-delivery roadmap, not a current gap.

### 8.2 Encryption in transit / at rest

- **In transit:** all Firebase SDK calls (Firestore, Storage, Auth, Functions) and all direct `fetch` calls to Google TTS/Gemini/Dicta/ARASAAC go over HTTPS/TLS — Firebase does not offer a non-TLS mode, and every third-party endpoint referenced in the CSP `connect-src` (`firebase.json:21`) is an `https://` origin. `[Assumption]` TLS 1.3 specifically is Google/Firebase's current platform default, not independently configured or verifiable from this repo — reasonable to state as "TLS in transit, version managed by the platform" rather than claiming a specific verified version.
- **At rest — Firestore/Storage plaintext-document data:** Google-managed, AES-256, server-side, transparent to the application (standard for all GCP-native services) — this covers *encryption at rest of the storage medium*, **not** encryption from Google's own view: Google (and, by extension, anyone who compromises the `co-board` GCP project's IAM) can read board/profile/child-metadata documents in plaintext, because nothing in this layer is end-to-end encrypted. This is an accurate, important distinction to state precisely for a DPIA: "encrypted at rest" (true, Google-managed) is not the same claim as "even the platform operator cannot read it" (false, for everything except media).
- **At rest — media, the actual E2EE layer, verified in detail:** `services/sync/crypto.ts` implements a real, non-trivial client-side encryption design:
  - **Algorithm:** AES-256-GCM (`ALGORITHM = 'AES-GCM'`, `KEY_LENGTH = 256`, `crypto.ts:12-13`).
  - **Key architecture:** a per-device root key (`getDeviceKey()`, `data/keyStore.ts:24-42`) generated via `crypto.subtle.generateKey(..., false, [...])` — **`extractable: false`** is the load-bearing security property: the raw key material can never be exported to JavaScript, even by the app's own code, which means an XSS payload that achieves arbitrary JS execution still cannot exfiltrate the key itself (it could still call `encrypt`/`decrypt` operations while the page is open, but cannot steal the key for offline use). This is a genuinely strong design choice, explicitly documented in the code's own comments as a deliberate improvement over an earlier JWK-in-`localStorage` scheme (`crypto.ts:2-3`, `keyStore.ts:10-11` — the `LEGACY_LS_KEY` migration path confirms this history and safely purges the old plaintext-exportable key from `localStorage` on first use, `keyStore.ts:44-64`).
  - **Per-file keys:** each media blob gets its own random 256-bit CEK, generated `extractable: true` **only** so it can be wrapped, then immediately wrapped (`wrapKey`) by the device key and never persisted unwrapped (`crypto.ts:102-118`). This is correct practice — the file's actual encryption key is never itself stored in reusable form outside the wrap.
  - **IVs:** two independent random 12-byte IVs per file (one for the key-wrap operation, one for the data encryption operation, `crypto.ts:110,114`) — correct; AES-GCM requires a unique IV per key-use, and reusing the CEK's IV for both the wrap and the data would be a real cryptographic error. This implementation avoids that mistake.
  - **Format versioning:** a 4-byte magic header (`'CB02'`) plus backward-compatible parsing of an older, deprecated PBKDF2-from-uid scheme (`deriveMediaKey`, `crypto.ts:80-95`, marked `@deprecated`, with a comment noting the old scheme's weakness: *"uid is not secret → anyone who knows it, including an admin, could have decrypted it"*). This shows real security maturity — the team fixed their own earlier design flaw while keeping safe backward-compat decryption.

  **The one design choice that undermines the whole scheme in a multi-device world: there is no key-sharing mechanism between devices or accounts.** Because the device key is non-extractable and never leaves the device (no export, no QR-pairing, no password-derived-key-escrow, no server-held wrapped-copy-per-authorized-device), **media encrypted on device A cannot be decrypted on device B — even for the same signed-in parent, let alone a different account granted `childAccess`.** Verified by tracing `decryptBlob()`'s only failure path (`crypto.ts:183-185`): a wrong-key `unwrapKey`/`decrypt` throws, is caught, and returns `null` — a *silent* failure with no user-facing distinction between "wrong key" and "corrupt file." **This is the most consequential technical finding in this document:**
  1. A family with a parent's phone + a parent's tablet, both signed into the same account with sync enabled, cannot share photos between those two devices today, despite the architecture otherwise supporting exactly that scenario.
  2. The child-sharing feature (§7.2.1), even with its other two bugs fixed, still wouldn't let a clinician view a shared child's synced photos — their device has no way to obtain a usable copy of the wrapping key.
  3. **This threatens reliability finding B-11 ("device loss = total data loss") in the opposite direction**: a family that enabled sync and replaced a lost device would find old encrypted photos undecryptable on the new device, because the new device generates its own fresh key on first use (`keyStore.ts:35-41`), unrelated to the old one. **Cloud sync of media today protects only "the Storage bucket still has the bytes" — not the ability to actually recover usable photos** — the opposite of what B-11's fix ("make sync default-on") is meant to deliver for this data type, unless this key-continuity gap closes first.

  **Recommendation:** before promoting "media is E2E-encrypted and synced" as a reliability/privacy selling point, close this gap. The standard pattern (used by e.g. Signal, iCloud Keychain, WhatsApp) is some form of **account-level key continuity**: derive or escrow a stable "family key" once (e.g., from a recovery phrase, or wrapped separately per authorized device via a device-linking flow at pairing time, or wrapped by a server-held key that is itself protected by the user's authenticated session rather than being fully unavailable to the server). Any of these preserves "the server operator cannot read the plaintext without user cooperation" while fixing multi-device continuity — full server-side key custody (which would let Co_Board decrypt everything) should specifically be avoided, as it would collapse the E2EE property the team clearly built deliberately.

- **What is explicitly NOT E2EE (state this plainly for the DPIA):** board text, cell labels, child names, ages, preferences, and all Firestore-resident data are **not** end-to-end encrypted — only media blobs are. Google-managed at-rest encryption applies, but Co_Board (and by extension anyone with sufficient GCP project access) can read this data in plaintext by design, because features like search, sync-merge, and the admin approval flow need to operate on it. This is a reasonable, common architecture choice — but it must be stated accurately in any privacy policy or DPIA rather than allowing "we use encryption" to be read as "everything is E2EE," which it isn't.

### 8.3 Data minimization, retention, right-to-erasure

- **Retention/deletion today is soft-delete only** — boards and children use an `archived`/`archivedAt` flag, never a hard delete (`domain/models.ts:113`, `childRepo.ts:31,95` — cross-reference `docs/01-audit.md` Part 15). There is no scheduled purge job found anywhere in `functions/src/` — archived data persists indefinitely with no documented retention window.
- **Account deletion path: does not exist.** Exhaustive grep for `deleteUser`, `deleteAccount`, and related terms across `app/src` and `functions/src` returns no matches. There is no in-app "delete my account and all associated data" flow, no Cloud Function that cascades a delete across `users/{uid}`, `users/{uid}/children/*`, `childAccess/*/members/{uid}`, and the corresponding Storage media. **For a product whose core promise is "trusted with the data of minors with disabilities," the absence of any erasure mechanism — self-service or admin-assisted — is a significant gap** for any right-to-erasure obligation (Israeli Privacy Law, GDPR Article 17, or simply good-faith trust with families). This is distinct from and more severe than the already-flagged "no trash UI" (B-20), which is about *recoverability*; this finding is about the *complete absence of a deletion capability at all*, recoverable or not.
- **Data export/portability:** `backupRepo.exportBackup()` (`data/backupRepo.ts:32-51`) produces a JSON envelope of boards, profiles, and settings — a genuine, working self-service export mechanism, and a reasonable *technical* answer to a GDPR Article 20 portability request for that data. **Gap:** the export does **not** include media blobs (`BackupData` interface, `backupRepo.ts:8-16`, has no media field) or voice recordings — a "full data export" claim would currently be false; only structured board/profile/settings data is portable today, not the photos/recordings that are arguably the most personally sensitive assets in the system. Recommend extending the export format to include decrypted media (the user's own device already holds the decryption key, so this is straightforward) before calling this "GDPR portability," and cross-reference §8.2's key-continuity gap, which is the same underlying limitation.
- **Right-to-access:** a user can already read their own Firestore documents directly via the app (rules permit `isOwnUid` read), and the backup export above serves as a structured "give me my data" mechanism for boards/profiles/settings. No formal "data subject access request" tooling exists (e.g., an admin-facing "export everything for uid X" button) — `[Assumption]` acceptable at current scale where the user *is* the primary consumer of their own export, but institutional customers (Part 9) will likely expect a formal DSAR process.

### 8.4 Consent flows

- **Current state is essentially: account signup only.** There is no explicit, itemized consent screen for: cloud sync (a `PrivacyToggle.tsx` component exists per ADR-0002 and defaults `syncEnabled = false` until the parent opts in — this **is** a real, working consent gate for sync specifically, verified by the ADR's explicit statement "פרטיות ברירת מחדל: syncEnabled = false עד שהורה מאשר מפורשות," `docs/adr-0002-sync.md:58`), but no consent flow exists for: sending utterance text to Google TTS (E-01, still an open TODO), sending topics to Gemini, sending words to Dicta, or for a **second parent's** consent when a family has more than one guardian.
- **Parent-consent-for-child:** by construction, the *only* signer on any Firebase account is an adult (no child accounts exist, §7.1) — so every action taken through the app is, in a narrow technical sense, "parent-authorized" because a parent's session performed it. This is not the same as *informed, itemized* consent to each specific data flow (TTS-to-Google, AI-to-Gemini, etc.) — those are currently bundled into "using the app" rather than disclosed and consented to individually.
- **Dual-parent / divorced-family scenarios: entirely unaddressed today.** There is exactly one Firebase account per child-owning family unit in the current model (the account that ran `createChild`, `childRepo.ts:98-120`, is the sole owner; a second parent can only be added via the same share-invite mechanism as a clinician, with no distinction between "co-parent with full rights" and "clinician with edit rights" — both currently map to the same `parent`/`clinician` `ChildAccessRole` values with no legal-guardianship semantics, no per-action consent gating, and no visibility to one parent over actions the other parent takes). This is squarely a Part 20 design gap, addressed in that section below.

## Part 9 – Multi-Tenant Assessment

### 9.1 The charter's core question: can one tenant accidentally access another tenant's data?

**Answer: No — for cross-*family* isolation, the rules as written correctly default-deny and were found to have no IDOR.** Every rule walked in §7.2 scopes access to `isOwnUid` or an explicit `hasChildAccess`/`ownsChildDoc` grant tied to `request.auth.uid`; there is no rule anywhere that grants blanket collection-level read across all users (the one `list` exception, `docs/firestore.rules:89-91`, is `admin`-only and used solely for the approval dashboard). This was verified both by reading every `match` block in `docs/firestore.rules` and by the existing rules-unit-test suite (`functions/test/rules.test.ts`), which specifically tests and confirms `"non-member cannot read another user children"` and `"non-owner cannot read shareInvites"`.

**However, the honest caveat: the current system has no concept of a "tenant" beyond an individual Firebase `uid`.** There is no `familyId`, `clinicId`, or `schoolId` field anywhere in the schema (`ChildRecord`, `Profile`, `UserRecord` — all reviewed, none has an organizational-tenant field). "Isolation" today is really **per-owner-account isolation with an opt-in per-child sharing grant bolted on** — which happens to be *sufficient* to answer "no, tenant A can't read tenant B's data" for the family-vs-family case the charter's core question is really asking, but is not yet a multi-tenant *architecture* in the sense the charter's "5,000 families + 500 clinics + 50 schools" scale scenario implies.

### 9.2 What "tenant" would mean for clinics and schools — gap analysis

- **Clinic (a therapist with 20 families):** would need 20 separate `childAccess` grants today, with no grouping entity, no "all my patients" dashboard (`ChildrenDashboard.tsx` only lists children the *signed-in uid owns*, confirmed in §7.2.1), and no clinic-level billing (B-02). Grants are also **permanent and unrevoked** (§9.3), so staff turnover has no clean offboarding path.
- **School (classroom-scoped staff):** the charter envisions "principal-approved, classroom-scoped" access — today's `staff` `ChildAccessRole` is a single flat role with no classroom grouping and no principal-approval workflow; grants are created by a parent, not a school administrator, so onboarding a classroom would require one separate parent-issued invite per child — operationally implausible at scale.
- **No audit log exists** (`grep -rn "auditLog\|auditEvent"` → no matches in `app/src`/`functions/src`). "Who accessed/changed which child's data, when" — a standard institutional procurement question — cannot be answered beyond Google Cloud's own platform-level audit logs, which aren't surfaced to customers `[Assumption]`.
- **No admin dashboard for an org to manage its own members** — the only admin surface (`AdminApprovalPanel.tsx`) is Co_Board's platform-level signup approval, not a clinic director managing their own therapists.

### 9.3 Remediation plan

1. **Fix the three §7.2.1 bugs first** — a multi-tenant model built on top of a sharing mechanism that doesn't functionally work end-to-end will inherit the same defects. Concretely: (a) align `AcceptInviteScreen.tsx`'s input validation/UI copy with the real 32-char hex code format; (b) extend the Firestore rule for `users/{uid}/{collection}/{document}` to also permit `hasChildAccess` for the specific `board`/`profile` collections scoped to a child the requester has a grant for (this requires restructuring the board storage path to be child-scoped rather than purely owner-uid-scoped — see Part 20's schema proposal, since the current `users/{ownerUid}/board/{boardId}` path has no `childId` in it at all, making a targeted rule fix non-trivial without a schema change); (c) solve the §8.2 multi-device key-continuity problem so that a legitimately-granted clinician's device can actually decrypt shared media.
2. **Introduce an explicit tenant/org entity** (`clinics/{clinicId}`, `schools/{schoolId}`) with its own member roster, separate from the per-child `childAccess` grants — Part 20 sketches this as `Organization` below.
3. **Add revocation** — a Cloud Function (`revokeChildAccess`) mirroring `acceptInvite`'s transactional pattern, callable by the owner, deleting the `childAccess/{childId}/members/{uid}` document; surface it in a UI list ("who currently has access to this child") per the DPIA finding in §8.6.
4. **Add an audit-log collection** — `auditLog/{eventId}` written by every Cloud Function that touches access grants or sensitive fields (`acceptInvite`, `approveUser`, the new `revokeChildAccess`), append-only from the client's perspective (client can query/read entries scoped to children they have access to; only Admin-SDK functions can write). Schema sketched in Part 20.
5. **Rules test additions** (extend `functions/test/rules.test.ts`): once (1)-(3) above land, add tests for: `"clinician CAN read a shared child's board after grant"`, `"clinician CANNOT read board after grant is revoked"`, `"staff role cannot write board (view-only)"`, `"clinic admin cannot read a child not assigned to their clinic"` (once org entities exist). None of these exist today — the current test suite (5 tests) only covers the negative "no privilege escalation" cases, not the positive "the feature actually grants what it should" cases, which is exactly the class of bug §7.2.1 found.
6. **Composite indexes** for future clinic/school dashboards (querying "all children accessible to uid X across owners") will require either a `childAccess` collection-group query (Firestore supports `collectionGroup('members').where('uid','==',X)` to find all grants for a given uid across every child) — **this is available today and is the missing piece for "list children shared with me,"** and should be the first client-side query added once the rules gap in (1)(b) is closed. `docs/01-audit.md` Part 4.4 already confirmed no missing-index problems exist for current query patterns; this new collection-group query pattern should be added to that inventory in a future audit pass.

## Part 20 – Family & Multi-Role Account Model (design proposal)

This section designs the target model the charter asks for, explicitly built as a **migration path on top of the existing `users/{uid}/children` + `childAccess` structure** (per the instruction not to rip-and-replace) — informed directly by the concrete gaps found in Parts 7-9 above, especially §7.2.1 (sharing doesn't work end-to-end) and §9.3 (no org entity, no revocation, no audit log).

### 20.1 Proposed entity model

**Guiding principle from §7.1: no child ever gets a Firebase Auth identity. Preserve this.** Everything below adds structure *around* the existing adult-only account model, not a child-login system.

```
users/{uid}                                  # unchanged — one Firebase Auth account per adult
  status: pending|approved|rejected           # unchanged (admin-approval gate, B-03)
  familyId?: string                           # NEW, additive — links this account to a Family entity

families/{familyId}                          # NEW top-level collection — the "billing entity" the charter asks for
  createdAt, createdBy: uid
  plan: 'free'|'family'|'clinic'|'school'      # billing tier (B-02 groundwork)
  billingCustomerId?: string                   # Stripe/Paddle customer ref, once B-02 is built

families/{familyId}/members/{uid}             # NEW — replaces the implicit "owner uid" concept
  role: 'parent'|'co-parent'|'admin'           # family-internal role, distinct from child-level ChildAccessRole
  addedAt, addedBy: uid
  consentRequiredFor?: string[]                # NEW — per-member flag for dual-custody mode (§20.3)

users/{uid}/children/{childId}                # UNCHANGED shape, additive field:
  childId, name, age, preferences, homeBoardId, archivedAt
  familyId: string                             # NEW — links the child to the Family, not just the creating uid

childAccess/{childId}/members/{uid}            # UNCHANGED shape, additive fields:
  childId, uid, role: 'parent'|'co-parent'|'clinician'|'staff'   # 'co-parent' is NEW
  grantedAt
  expiresAt?: number                           # NEW — time-bound grants for clinician/staff (charter requirement)
  scope?: string                               # NEW — e.g. a classroomId for staff, "classroom-scoped" per charter
  revokedAt?: number                            # NEW — soft-revoke, keeps an audit trail instead of hard-deleting

organizations/{orgId}                          # NEW top-level collection — the clinic/school tenant entity (Part 9)
  type: 'clinic'|'school'
  name, createdAt, adminUid
  plan: 'clinic'|'school'                       # billing tier

organizations/{orgId}/staff/{uid}              # NEW — roster of an org's members
  role: 'admin'|'therapist'|'staff'
  addedAt

organizations/{orgId}/childLinks/{childId}     # NEW — which children this org currently has a relationship with
  linkedBy: uid (must also hold a childAccess grant for childId — enforced in the granting Cloud Function)
  linkedAt, expiresAt?, revokedAt?

auditLog/{eventId}                             # NEW — append-only, Admin-SDK write only (Part 9 remediation #4)
  timestamp, actorUid, action: 'grant'|'revoke'|'delete'|'consent'|'export'|'invite_created'|'invite_redeemed'
  targetChildId?, targetFamilyId?, targetOrgId?
  metadata: Record<string, unknown>              # e.g. { role: 'clinician', expiresAt: ... } — never raw child content

voiceConsent/{consentId}                        # NEW, per Stage E's §14.3 plan — included here for schema consistency
  speakerId, consentGivenBy: uid, consentType: 'self'|'guardian', timestamp, revokedAt?
```

**Migration path (no rip-and-replace):**
1. **Additive fields first** (`familyId` on `users` and `children`, `expiresAt`/`scope`/`revokedAt` on `childAccess` members) — all optional, all backward-compatible with existing documents (mirrors the codebase's own stated DB-migration philosophy, "additive-only," already used for IndexedDB per `docs/01-audit.md` Part 4.1).
2. **Backfill script** (one-time Cloud Function or admin script): for every existing `users/{uid}` with at least one child, create a `families/{familyId}` document and a `families/{familyId}/members/{uid}` entry with `role: 'parent'`, then stamp `familyId` onto their existing `children/*` docs. Zero data loss, zero schema break — existing `users/{uid}/board/...` sync paths, `childAccess` grants, and rules continue working unmodified during and after this backfill.
3. **New rules, additive, not replacing:** add a rule branch that also permits `hasChildAccess`-based read/write on `users/{uid}/{collection}/{document}` **scoped specifically to `board`/`profile` collections tied to a child the requester has an active (non-revoked, non-expired) grant for** — this is the fix for §7.2.1(1), and is the one change here that requires a schema addition (a `childId` field on board documents, so the rule can join `board` → `childId` → `childAccess`) rather than being purely additive at the rules layer alone.
4. **Organizations are opt-in, additive, and don't touch the family model** — a clinic signs up independently; its `childLinks` are created only through the *existing* consent chain (a family must first grant `childAccess` to a specific therapist uid via the existing/fixed invite flow; that therapist's org can then optionally associate the link with their org roster for dashboarding purposes). This preserves "families own the consent decision" as the root of trust, with orgs layered on top for operational convenience (shared dashboards, billing) — not as an independent grant of access.

### 20.3 Dual-custody / divorced-parents mode

**Design:** a `families/{familyId}` can have **two or more** `parent`/`co-parent` members. For a defined set of **sensitive actions** — the charter's examples (voice cloning, child deletion, new-adult invites) plus this review's addition (revoking another parent's invite) — require **dual consent** before the action takes effect:

1. Parent A initiates a sensitive action (e.g., delete a child's profile, invite a new clinician).
2. The action is written to a `pendingConsent/{requestId}` document (not yet applied): `{ familyId, childId, action, initiatedBy, requiredApprovers: [...], approvals: {}, createdAt, expiresAt }`.
3. Every other `parent`/`co-parent` member is notified (once push/email exists per Part 12/13's roadmap; an in-app banner on next login is a reasonable interim).
4. The action executes (via a Cloud Function mirroring `acceptInvite`'s transactional pattern) only once all `requiredApprovers` have approved, or via an explicit, audit-logged unilateral-override path for urgent safety cases `[Assumption — product decision needed]`.
5. **Inactive by default:** a single-parent family (today's universal case) behaves exactly as today — the gate only activates once a family has ≥2 `parent`-tier members, keeping this additive and low-friction for the majority.
6. This mechanism deliberately doesn't try to distinguish "divorced" from "trusting co-parents" — the same dual-consent flow serves both without the app adjudicating custody status, which is out of scope for a product to determine. `[TBD — counsel]` whether any jurisdiction imposes a specific obligation beyond "reasonable technical measures against unilateral action."

### 20.4 Invite flow design (building on the existing one-time codes)

The existing mechanism (`createShareInvite`/`acceptInvite`, §7.2) is architecturally sound (128-bit entropy, atomic transactional redemption, server-side expiry/role revalidation) — the fix list is UI/rules/schema alignment (§9.3 remediation #1), not a redesign. On top of the fixed mechanism, add:

1. **Time-bound by default, not just at the invite-redemption stage.** Today, the invite itself expires in 48 hours, but the resulting `childAccess` grant is permanent. Change: every `clinician`/`staff` grant created via invite gets a default `expiresAt` (e.g., 12 months, configurable by the inviting parent at invite-creation time) — matching the charter's explicit "time-bound, revocable" requirement for therapist links. `parent`/`co-parent` grants remain non-expiring (a family relationship isn't time-boxed).
2. **Visible grant list + one-click revoke** (§9.3 remediation #3) — a settings panel (natural extension of the existing `AccessSettingsPanel.tsx`, already flagged in `docs/01-audit.md` B-05 as a file that "will only get more complex once Part 20 lands, refactor before adding roles" — this review concurs with that sequencing) listing every current `childAccess` member for a child, their role, grant date, expiry, and a revoke button calling the new `revokeChildAccess` Cloud Function.
3. **Renewal, not just expiry.** A clinician whose 12-month grant is expiring should get a renewal prompt to the parent, rather than silently losing access mid-treatment — this is a product/UX detail to hand to the UX Auditor (Stage C) once this schema exists.
4. **Audit every invite lifecycle event** (`invite_created`, `invite_redeemed`, `grant_revoked`, `grant_expired`) into `auditLog/{eventId}` (§20.1 schema) — directly answers the DPIA's §8.6 finding about unrevoked/untracked access.

### 20.5 Audit-log schema and retention

Already sketched in §20.1 (`auditLog/{eventId}`). Retention recommendation: keep indefinitely for `grant`/`revoke`/`delete`/`consent` event *types* (these are exactly the events an institutional auditor or a family dispute would need to reconstruct months later), but cap `invite_redeemed`/routine read-access events at a rolling 12-month window if volume becomes a cost concern (Firestore write costs are negligible at this event rate per `docs/01-audit.md` Part 16's cost model — no urgency to aggressively prune). Access to `auditLog` entries should be readable by: the family's own `parent`/`co-parent` members (scoped to their own `familyId`/`childId`s), an org's own admin (scoped to their `orgId`), and platform admin — never cross-family, mirroring every other access pattern in this document.

### 20.6 Billing model sketch (references B-02)

- **Family plan:** per-family monthly/annual subscription, `families/{familyId}.plan` field drives feature gating (e.g., number of children, AI board-generation quota, voice-recording storage cap). `billingCustomerId` links to Stripe/Paddle (B-02's recommended providers). Free tier remains viable for a single child/basic features, consistent with the freemium positioning already implied in the PRD per `docs/01-audit.md` Part 16.
- **Clinic plan:** per-organization subscription, priced per-active-therapist-seat or per-linked-family count `[Assumption — pricing model needs product/sales input, out of scope for this technical document]`; `organizations/{orgId}.plan` drives access to the clinic dashboard (the "list all my patients" view that §7.2.1/§9.2 identified as currently missing) and bulk-invite tooling.
- **School plan:** similar to clinic, but priced per-classroom or per-enrolled-student-count, with the classroom-scoped `staff` grants (§20.1) as the enforcement mechanism for "only see your classroom's children."
- **Cross-reference:** this billing sketch is intentionally light — B-02 already flagged the complete absence of billing code as a Critical, Large-effort item owned by Architect+Analyst; this section exists only to confirm the Part 20 schema (`families`/`organizations` with a `plan` field) is billing-model-compatible, not to re-scope B-02's implementation.
