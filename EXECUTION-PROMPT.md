# פרומפט ביצוע — בניית אפליקציית ה-AAC ("לוח תקשורת")

> **איך להשתמש:** פתח/י סשן חדש בתיקיית הפרויקט והדבק/י את כל התוכן שמתחת לקו. הפרומפט מניח שהקבצים `PRD/PRD-he.md`, `HANDOFF.md` ומסמכי המחקר נמצאים ב-repo.

---

## התפקיד והמטרה
אתה מהנדס/ת תוכנה בכיר/ה שבונה את אפליקציית ה-AAC העברית "לוח תקשורת". **מקור האמת:** `PRD/PRD-he.md` + `HANDOFF.md` — קרא אותם **לפני** כל פעולה, ואל תגזור את המערכת מחדש מהקוד אם הם מספיקים.

**המשימה:** לבנות את ה-**MVP** (PRD §11.1) עבור **Android + Web (PWA)** קודם, עברית-ראשונה (RTL מלא, ניקוד, TTS עברי), ולהמשיך לפי מפת-הדרכים (PRD §12). iPad/iOS בפאזה 2.

לפני כתיבת קוד: הצג תוכנית קצרה + הכרעת stack + אישור גישה לשירותים חיצוניים (TTS/נקדן/סמלים), ואז התקדם milestone-by-milestone עם commits תכופים.

## עקרונות עבודה (חובה — לא לוותר)

### 1. האצלה מדורגת לתתי-סוכנים לפי גודל המשימה
| משימה | סוכן | מודל |
|-------|------|------|
| חיפוש / איתור קוד / חקירה | investigator / Explore | **Haiku** (זול) |
| תיקון קטן, 1–2 קבצים | builder | **Sonnet** |
| פיצ'ר / refactor מורכב | **Plan (Opus) → builder (Sonnet) → reviewer (Opus)** | בהתאם |
| מחקר / תיעוד | general-purpose | Sonnet |

- הצינור המלא Opus→Sonnet→Opus רץ **רק** לעבודה לא-טריוויאלית. **לעולם לא 3 סוכנים קרים לתיקון שורה אחת.**
- אל תדלג על אימות-Opus הסופי בשינויים מורכבים.
- כל סוכן מתחיל "קר" וטוען context מחדש — אל תפריז בהאצלה; שקול עלות טוקנים מול תועלת.

### 2. מקביליות
- אגד קריאות-tool עצמאיות בהודעה אחת.
- הרץ פיצ'רים בלתי-תלויים במקביל (worktrees נפרדים) ומזג.
- כשצעד חוסם רץ (install/build/download) — אל תבטל; הרץ תת-סוכן להכנה שאינה תלויה במשאב החסום.

### 3. סוכן-משפר רץ ברקע כל הזמן (Continuous Improver)
- לאורך כל הפיתוח, החזק **סוכן reviewer/improver שרץ במקביל** ומבקר ומשפר ברציפות את התוצרים האחרונים: איכות קוד, נכונות RTL, דיוק ניקוד/TTS עברי, נגישות, כיסוי בדיקות, אבטחה ופרטיות.
- הסוכן פותח משימות שיפור (TaskCreate) וממוקד ב**תוצרים משמעותיים** — לא בקבצים טריוויאליים (לשמירת טוקנים).
- כל אזהרה על פגיעה באינווריאנט (למשל הזזת מילות ליבה) עוצרת ומועלית מיד.

### 4. תיעוד קפדני (פרוטוקול HANDOFF)
- אחרי **כל** שינוי שמשנה התנהגות: עדכן את שורת-האינווריאנט התואמת ב-`HANDOFF.md` + מסמך-הפירוט הרלוונטי + הוסף שורת changelog — **באותו commit**.
- המר תאריכים יחסיים למוחלטים.
- שמור את `HANDOFF.md` כ-~80% כיסוי; פירוט עמוק → מסמך מקושר ב-`docs/`, לא משוכפל.

### 5. בדיקות (שער CI חוסם)
- גישת test-first ככל האפשר. **Unit** + **Widget/Integration**.
- **חבילת QA עברית** (PRD נספח C): ניקוד, הומוגרפים, זכר/נקבה, יחיד/רבים, משפטים טבעיים, שמות פרטיים, סלנג, ערבוב שפות.
- **נגישות:** WCAG 2.1 AA, גודל מטרה ≥~1.5 ס"מ, ניגודיות, מצב נעול.
- **Offline-first:** בדיקות שכל מסלול קריטי עובד ללא רשת.
- **ביצועים:** משוב ויזואלי <100ms, תחילת TTS אופליין <300–500ms, מעבר לוח <200ms.
- **סנכרון:** בדיקות התנגשות רב-מכשירית ושחזור גרסאות.

### 6. יעילות טוקנים ועלות
- העדף עריכות ממוקדות על קריאה מחדש של קבצים שלמים. אל תפתח סוכן יקר למשימה זולה.

## אינווריאנטים קריטיים (מתוך HANDOFF §4 — אסור להפר)
1. מילות ליבה **לא זזות** ממיקומן לאורך התפתחות הילד (Motor Planning). אזהרה לפני כל שינוי פריסה.
2. **Offline-first:** תקשורת/ניווט/TTS בסיסי/סמלים עובדים ללא רשת.
3. ניקוד: cache מקומי + תיקון ידני נשמר; ללא תלות ברשת בשימוש חוזר.
4. פרטיות: נתוני ילדים — אנליטיקה כבויה כברירת מחדל/מקומית; הורה שולט בהעלאה. עמידה ב-GDPR/COPPA/חוק הגנת הפרטיות.
5. מצב ילד נעול; מעבר לעריכה רק בקוד מטפל (RBAC).
6. RTL מלא בכל מסך.
7. שינוי AI/אוטומטי מכבד עקביות מיקום.

---

## ארכיטקטורה (מתוך PRD §9 / HANDOFF §3)
- **Client** חוצה-פלטפורמות (קוד משותף Android + Web → iOS). **מומלץ Flutter**; אם בוחרים אחרת — נמק ועדכן HANDOFF.
- שכבות: Presentation (RTL, child/adult modes) → Domain (boards/profiles/navigation/permissions) → Services (TTS, Nikud, Media, Symbols, Analytics) → Data (offline-first local DB → async sync → Cloud).
- **TTS היברידי:** מנוע אופליין מקומי + קולות פרימיום מקוונים (Almagu מועדף). **זה הסיכון הטכני המרכזי** — טפל בו מוקדם (spike ב-M0/M1).
- **ניקוד:** Nakdan API + cache. **סמלים:** ARASAAC בסיס → SymbolStix/PCS ברישוי. **מדיה:** מצלמה + הסרת רקע on-device + WebP.

## סדר עבודה (Milestones) — כל אבן-דרך עם Definition of Done
- **M0 — Scaffold & Foundations:** הכרעת stack; init פרויקט; CI (lint+test+build); מבנה תיקיות; **spike TTS+ניקוד אופליין בעברית**; אישור גישה לשירותים.
- **M1 — Data & Profiles:** מודל נתונים (לוחות/תאים/פרופילים); DB מקומי offline-first; פרופילי ילד מרובים; הרשאות (RBAC); מצבי ילד/מבוגר.
- **M2 — Communication Core:** רינדור לוח (RTL, גדלי גריד דינמיים, קידוד פיצג'רלד); ניווט (בית/חזור/מחק/נקה/עוד + קישור תא); שורת משפט + "דבר"; **TTS עברי (אופליין בסיס)** + **ניקוד אוטומטי + תיקון ידני**; יעדי חביון.
- **M3 — Builder & Symbols:** Builder (drag-drop, undo/redo, preview, **עריכה קבוצתית**); ספריות סמלים; צילום/העלאה + חיתוך + הסרת רקע; ייצוגים מרובים למילה.
- **M4 — Adaptivity:** הסתרה/חשיפה הדרגתית; מצב נעול (Guided Access); הגדרות גישה (Dwell, Activate-on-Release, מניעת מגע כפול); Onboarding הורה.
- **M5 — Persistence:** גיבוי/שחזור גרסאות; סנכרון ענן בסיסי דו-כיווני; תיעוד שימוש בסיסי (ניתן לכיבוי).
- **M6 — Hardening:** נגישות (WCAG), ביצועים, חבילת QA עברית מלאה, אבטחה/פרטיות, ייצוב לפני שחרור.

לכל פיצ'ר ב-MVP ראה רשימת החובה ב-PRD §11.1 ואת הדרישות הפונקציונליות (§7, FR-001…FR-024,027).
